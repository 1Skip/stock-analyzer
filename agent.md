## 沟通与协作偏好

- 日常沟通默认使用中文，包括需求探讨、决策表达、项目偏好、文档讨论和执行反馈。
- 技术搜索、技术报错、API 问题、库文档检索、错误关键词等可以保留英文，以便获得更准确的技术结果。
- 回答直奔主题，少做不必要确认，以效率和结果为导向。
- 如果存在更简单、更稳妥的方法，应主动提出。

## 不确定性处理与编码前思考

- 先理解目标、约束、现有上下文，再动手修改。
- 对不确定事项不要假设，不要隐藏困惑；需要时明确说明假设、呈现权衡，并提出需要澄清的问题。
- 如果存在多种合理解释，不要默默选择其中一种，应说明可能路径和取舍。
- 对多步骤任务，先给出简短计划，并为每一步定义验证方式。

## 简洁优先

- 用最少的代码解决问题，不做过度工程。
- 不添加需求之外的功能。
- 不为一次性代码创建抽象。
- 不添加未要求的“灵活性”或“可配置性”。
- 不为极不可能发生的场景堆叠复杂错误处理。
- 如果 200 行代码可以清楚地写成 50 行，应优先简化。
- 检验标准：资深工程师如果会觉得方案过于复杂，就继续简化。

## 精准修改与提交规范

- 只修改完成用户请求必须修改的内容。
- 不顺手“改进”相邻代码、注释或格式。
- 不重构没有问题的代码。
- 匹配现有代码风格，即使有其他个人偏好的写法。
- 文件体积、变量命名、接口边界都要保持简洁清晰。
- 风格一致、命名清晰、接口明确。
- 如果发现无关死代码，可以指出，但不要擅自删除。
- 如果自己的改动造成无用导入、变量、函数或孤儿代码，应清理这些由本次改动产生的内容。
- 每一行修改都应能直接追溯到用户请求。
- 提交前应完成必要检查，包括格式、测试、构建或项目约定的验证命令。

## 安全红线

- 密码、个人令牌、私密配置不得硬编码进源码。
- API Key、支付密钥、短信密钥、数据库密码等必须通过环境变量或安全配置读取，不能写死在代码中。
- `.env` 文件不得提交仓库，应进入忽略规则。
- 前端代码不得包含任何服务端密钥或敏感配置。
- 如果发现疑似密钥、令牌或敏感信息，应立即提醒并避免继续扩散。

## 目标驱动与测试闭环

- 将模糊任务转化为可验证目标，成功标准应明确、可检查。
- “修复 bug”应转化为“编写或确认能复现问题的测试，然后让测试通过”。
- “添加验证”应转化为“为无效输入定义验证标准，并用测试或运行结果确认”。
- “重构 X”应转化为“确保重构前后关键测试都通过”。
- 优先采用 TDD 或测试优先策略：先明确功能验证标准，再让实现围绕标准展开。
- 执行必要运行命令，保留真实反馈；不要掩饰错误。
- 修复后必须重新验证，通过后才算闭环完成。
- 如果无法运行测试或验证命令，应明确说明原因和剩余风险。
